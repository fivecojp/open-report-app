オープン報告アプリ システム仕様書
1. システム概要
店舗スタッフが出勤時に「店舗のオープン報告（写真付き）」を行うためのWebアプリケーション。
新しいデータベース環境（Supabase）へのデータ蓄積と、既存システム（Google Workspace: ドライブ/スプレッドシート）へのデータ送信を同時に行うブリッジシステムとして機能する。

2. 技術スタック
フロントエンド・バックエンド: Next.js (App Router, React)

スタイリング: Tailwind CSS (ダークテーマ: #0f1923, #182030, アクセント #38c9a0)

データベース: Supabase (PostgreSQL)

外部連携: Google Apps Script (GAS) Webhook

デプロイ・ホスティング: Vercel

ソースコード管理: GitHub

3. 環境変数 (Environment Variables)
ローカルの .env.local および Vercel の Environment Variables にて管理。

NEXT_PUBLIC_SUPABASE_URL: SupabaseのプロジェクトURL

NEXT_PUBLIC_SUPABASE_ANON_KEY: Supabaseの公開キー（フロントエンド用）

SUPABASE_SERVICE_ROLE_KEY: Supabaseの管理者キー（サーバー側専用・RLS回避用）

GAS_WEBHOOK_URL: Google Apps Script のデプロイURL (https://script.google.com/.../exec)

4. 画面・UI仕様 (app/page.tsx, components/OpenReportForm.tsx)
アクセスURL: /?store=[店舗のUUID] (店舗IDが必須)

ヘッダー: Supabaseから取得した店舗名を表示。

報告者選択: clock_in_reports テーブルから「本日出勤しているスタッフ」を取得し、プルダウンで選択させる。

カメラUI: <input type="file" capture="environment" /> を使用し、スマホの背面カメラを直接起動。

画像圧縮: Canvas API を使用し、最大幅 800px、JPEG (品質 0.82) の Base64 形式にフロントエンドで圧縮。

履歴表示: フォーム下部に、過去30日分のオープン報告履歴をリスト表示。遅延がある場合は警告色（赤）、画像URLがある場合はリンクを表示。

5. データベース仕様 (Supabase)
利用する主要テーブルと役割。

stores (店舗マスタ)

store_id (uuid): URLパラメータで渡されるID

store_name (text): 画面表示名

staffs (スタッフマスタ)

staff_id (uuid)

staff_name (text)

clock_in_reports (出勤打刻)

store_id, staff_id, work_date (date) -> 報告者のプルダウン生成に使用。

store_business_hours (店舗営業時間マスタ)

store_id, weekday_type, open_time (time) -> 遅延計算の基準値として使用。

actual_business_hours (オープン報告の記録先)

store_id (uuid)

business_date (date): 営業日 (YYYY-MM-DD)

opened_at (timestamp): オープン打刻時刻

opened_by_staff_id (uuid): 報告者

open_delay_mark (boolean): 遅延フラグ (遅延分数が1以上の場合は true)

delay_minutes (int4): 定刻からの遅延分数 (手動追加カラム)

image_url (text): GAS経由で保存されたGoogleドライブのURL (手動追加カラム)

6. 処理フロー (app/actions/report.ts - Server Action)
フロントエンドから送信ボタンが押された後のサーバー側の流れ。

定刻取得と遅延計算:

store_business_hours から対象店舗・本日の open_time を取得。

現在時刻と比較し、delay_minutes (遅延分数) を計算。0より大きければ open_delay_mark を true にする。

Supabase 仮保存 (INSERT):

actual_business_hours に報告データ（時刻、スタッフIDなど）を INSERT し、作成されたレコードのIDを取得する。

ID変換 (Mapping):

Supabaseの store_id (UUID) を、既存のGAS/スプレッドシートが認識できる数値ID (1, 2, 3...) に変換する (STORE_ID_MAP を使用)。

GASへの送信 (fetch POST):

エンドポイント: process.env.GAS_WEBHOOK_URL

ペイロード: { storeId(数値), storeName, reporter(名前), image(Base64) }

GASからの応答と Supabase 本保存 (UPDATE):

GAS側でGoogleドライブへの画像保存・スプレッドシートへの記録が行われ、画像のURLが返却される ({ status: "ok", imageUrl: "..." })。

取得した image_url、計算した delay_minutes、open_delay_mark を、手順2で作成した Supabase のレコードに UPDATE する。

7. 今後の改修時の注意点
IDの不一致に注意: Supabaseは UUID (文字列)、既存GASのシステムは 数値形式 で店舗を管理しているため、report.ts 内の STORE_ID_MAP による変換ロジックが必須。新店舗が増えた場合はこのマップへの追記が必要。

RLS (Row Level Security): Supabaseの権限設定により、データが取得・書き込みできないエラーが起きやすい。本アプリでは utils/supabaseServer.ts にて SUPABASE_SERVICE_ROLE_KEY を使用し、サーバー側（Server Actions）で安全にRLSをバイパスする設計としている。

画像の扱い: Next.jsのサーバー（Vercel）を経由して大きなBase64文字列を送るため、Vercelのタイムアウト（通常10〜15秒）に注意。フロントエンドでのCanvas圧縮（800px等）の処理は外さないこと。
